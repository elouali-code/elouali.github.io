Data Engineering / BI

Pipeline Meteo de Bout en Bout

Conception d'un pipeline data engineering complet, depuis l'ingestion API jusqu'a la visualisation BI, avec une organisation en couches inspiree des pratiques data lake / warehouse.

RoleData Engineer
TypeProjet Personnel
NiveauOriente Production
StackPython, Pandas, DuckDB, SQLite, Metabase

Contexte

Le but du projet etait de construire un pipeline de donnees complet, pas seulement un script d'extraction. Les donnees meteo sont un bon cas d'usage car elles combinent ingestion API, normalisation de formats, gestion de villes, horodatage, controles qualite et restitution analytique.

J'ai structure le projet comme un mini systeme data: une couche raw pour conserver la donnee source, une couche transformee pour nettoyer et standardiser, puis une couche warehouse pour alimenter les indicateurs BI.

Architecture data

  • Ingestion batch en Python depuis une API meteo publique avec gestion des villes et des timestamps.
  • Stockage des reponses brutes dans une couche raw afin de pouvoir rejouer ou auditer les transformations.
  • Nettoyage avec Pandas: typage des colonnes, normalisation des unites, controle des valeurs manquantes.
  • Utilisation de DuckDB pour preparer des vues analytiques et tester les aggregations rapidement.
  • Chargement dans SQLite comme entrepot local simple, portable et facile a connecter a Metabase.
  • Creation de dashboards pour suivre temperature, humidite et comparaisons par ville.

Qualite & exploitation

  • Separation des responsabilites entre ingestion, transformation, chargement et visualisation.
  • Organisation du depot pour pouvoir relancer chaque etape sans melanger les donnees brutes et transformees.
  • Preparation de tables finales lisibles par un outil BI, plutot que de connecter directement les donnees brutes.
  • Choix d'une stack locale volontairement legere pour montrer la logique data engineering sans dependance cloud.

Resultats

Pipeline complet

La chaine couvre l'ensemble du cycle: API, raw, transformation, warehouse local et dashboard BI.

Dashboards exploitables

Les indicateurs finaux permettent de comparer les villes et de suivre les tendances meteo dans le temps.

Approche modulaire

Chaque etape peut evoluer independamment: nouvelle source API, nouvelle transformation ou nouveau dashboard.

Ameliorations possibles

  • Ajouter une orchestration planifiee avec Airflow, Prefect ou GitHub Actions.
  • Ajouter des tests de qualite de donnees pour detecter automatiquement les valeurs incoherentes.
  • Remplacer SQLite par un warehouse cloud si le volume ou le nombre d'utilisateurs augmente.
