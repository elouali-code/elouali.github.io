Migration Architecture Serverless O(1)

Refonte d'une architecture de consultation data afin de remplacer un parcours couteux de fichiers par un acces direct, previsible et industrialise autour de services AWS serverless.

RoleConception Cloud & Data
ContexteAirbus
DomaineArchitecture Data Serverless
StackDynamoDB, Lambda, SQS, S3, AWS CDK

Contexte

Le systeme initial reposait sur des fichiers stockes dans S3. Pour repondre a une requete, l'application devait identifier puis parcourir un volume de donnees trop large par rapport au besoin reel. Ce fonctionnement etait acceptable pour de l'archivage ou de l'analyse batch, mais moins adapte a un usage applicatif avec besoin de reponse rapide.

L'objectif etait de conserver la robustesse du stockage data tout en ajoutant une couche de consultation plus efficace, plus maintenable et plus simple a deployer dans differents environnements.

Architecture cible

  • Creation d'un modele DynamoDB base sur les cles de consultation metier les plus frequentes.
  • Conservation de S3 comme couche source / stockage durable pour les donnees brutes ou historisees.
  • Utilisation de Lambda pour exposer une logique stateless de lecture, transformation legere et controle des erreurs.
  • Ajout de SQS pour decoupler certains traitements et absorber les variations de charge.
  • Description de l'infrastructure avec AWS CDK afin de versionner, relire et redeployer l'architecture proprement.

Demarche

  • Analyse des chemins de lecture existants pour identifier les requetes lentes et les patterns repetitifs.
  • Definition d'une strategie de cle DynamoDB adaptee aux acces applicatifs, sans chercher a reproduire le modele fichier.
  • Separation claire entre stockage, file d'evenements, logique applicative et infrastructure as code.
  • Validation progressive avec des jeux de donnees representatifs afin de verifier le comportement et la simplicite d'exploitation.

Resultats

Acces plus directs

Les lectures applicatives ne dependent plus d'un parcours large de fichiers: la donnee cible est adressee par cle.

Scalabilite native

Le recours a DynamoDB, Lambda et SQS reduit la gestion serveur et facilite l'absorption des variations de charge.

Infrastructure reproductible

La definition CDK rend les environnements plus faciles a relire, deployer et maintenir.

Points d'attention

  • Le modele DynamoDB impose de bien connaitre les patterns d'acces avant de definir les cles.
  • La solution ne remplace pas le data lake: elle ajoute une couche optimisee pour les usages applicatifs.
  • Les erreurs et cas limites doivent etre geres explicitement dans les Lambdas pour garder un comportement previsible.
