Étude de Cas : Migration vers une Architecture de Données Serverless sur AWS

Conception et implémentation d'une solution data moderne pour résoudre les problèmes de performance et de scalabilité chez Airbus.

  • RôleArchitecte Cloud & Développeur Python
  • EntrepriseAirbus
  • Durée3 mois
  • TechnologiesAWS (DynamoDB, Lambda, SQS, API Gateway), Python, IaC (AWS CDK)
❮ ❯

Le Contexte : Un Stockage Inadapté

L'architecture initiale pour les données de produits de simulation reposait sur un unique fichier Parquet stocké sur Amazon S3. Cette approche, bien qu'efficace pour des analyses en batch, était totalement inadaptée aux accès transactionnels fréquents. Chaque mise à jour, même pour une seule donnée, forçait la lecture et la réécriture de l'intégralité du fichier, un processus lourd et coûteux.

La Problématique : Une Complexité en O(N)

Le problème fondamental de l'ancienne architecture résidait dans sa complexité algorithmique. Les opérations de lecture et d'écriture avaient une complexité temporelle linéaire, notée O(N) , où N est le nombre total de produits. Cela signifie que le temps de réponse se dégradait proportionnellement à l'augmentation du volume de données, rendant le système non scalable et voué à l'échec à long terme.

Ma Solution : Une Architecture Serverless en O(1)

Pour garantir des performances constantes quelle que soit la charge, j'ai conçu et implémenté une architecture serverless dont les opérations clés s'exécutent en temps constant, soit une complexité de O(1) . La démarche a d'abord été validée par une Preuve de Concept (PoC) avant d'être entièrement codée via l'Infrastructure as Code (IaC) avec AWS CDK.

  • Flux d'Écriture Asynchrone : Pour garantir la fiabilité, une requête de mise à jour est postée dans une file d'attente Amazon SQS. Ce découplage permet au client de recevoir une réponse immédiate. En parallèle, une fonction AWS Lambda "Consumer" est déclenchée par la file pour exécuter une opération `PutItem` atomique sur une table Amazon DynamoDB. Cette opération sur la clé primaire a une complexité de O(1) .
  • Flux de Lecture Synchrone : Une API Gateway expose un point de terminaison HTTP sécurisé. Un appel à cette API déclenche une fonction AWS Lambda "Getter" qui exécute une opération `GetItem` sur DynamoDB, en utilisant également la clé primaire. Cette lecture, d'une complexité de O(1) , retourne la donnée requise en quelques millisecondes.

Résultats & Impact

Performance O(1)

La latence des opérations est passée de plusieurs secondes à quelques millisecondes, indépendamment du volume de données.

Scalabilité Native

L'architecture peut gérer des milliers de requêtes simultanées grâce aux services managés AWS, sans intervention.

Optimisation des Coûts

Le modèle serverless "pay-as-you-go" a éliminé les coûts fixes superflus liés au traitement de fichiers volumineux.

Déploiement Fiable (IaC)

L'infrastructure, entièrement définie en code avec AWS CDK, est versionnable, reproductible et facile à maintenir.
