Projet : Migration d'une Architecture de Données vers AWS Serverless

Optimisation des performances et des coûts pour la plateforme Core S&T chez Airbus.

Contexte et Problématique

[cite_start]L'architecture initiale de stockage des produits de simulation reposait sur un unique fichier Parquet sur Amazon S3. [cite: 371] [cite_start]Avec l'augmentation du volume de données, cette approche montrait des limites critiques : chaque mise à jour, même mineure, nécessitait de lire, modifier et réécrire l'intégralité du fichier. [cite: 373] [cite_start]Ce processus entraînait des latences importantes et une complexité algorithmique linéaire (O(N)), ce qui n'était ni scalable ni efficient en termes de coûts. [cite: 18, 375]

Ma Solution et mes Actions

Pour résoudre cette problématique, j'ai conçu et implémenté une nouvelle architecture entièrement serverless sur AWS. La solution s'articule autour de deux flux distincts pour optimiser les performances :

    [cite_start]
  • Flux d'Écriture Asynchrone : Pour garantir la fiabilité, les requêtes de mise à jour sont envoyées à une file d'attente Amazon SQS. [cite: 54] Une fonction AWS Lambda consomme ensuite les messages de manière asynchrone et effectue une écriture ciblée (opération PutItem) dans une table Amazon DynamoDB. [cite_start]Ce découplage assure la résilience du système et une confirmation de réception instantanée. [cite: 54, 389, 390, 391]
    • [cite_start]
  • Flux de Lecture Synchrone : Pour des consultations rapides, une API Gateway expose un point de terminaison qui déclenche une autre fonction Lambda. [cite: 55] [cite_start]Celle-ci effectue une lecture directe et ultra-performante (opération GetItem) sur DynamoDB en utilisant la clé primaire, retournant la donnée avec une latence minimale. [cite: 55, 395]

[cite_start]L'intégralité de cette infrastructure (table DynamoDB, Lambdas, file SQS, API Gateway, rôles IAM) a été provisionnée de manière automatisée et reproductible via l'AWS Cloud Development Kit (CDK), en suivant les principes de l'Infrastructure as Code (IaC). [cite: 56]

Résultats et Impact

Cette migration a eu un impact majeur sur la performance et l'efficacité de la plateforme :

    [cite_start]
  • ✅ Performance Drastiquement Améliorée : La latence des opérations est passée de plusieurs secondes à quelques millisecondes. [cite: 436]
    • [cite_start]
  • ✅ Complexité Réduite : La complexité des opérations de lecture/écriture a été réduite de linéaire **O(N)** à constante **O(1)**. [cite: 453]
    • [cite_start]
  • ✅ Scalabilité Native : L'architecture serverless peut gérer des milliers de requêtes sans intervention manuelle. [cite: 437]
    • [cite_start]
  • ✅ Optimisation des Coûts : Le modèle "pay-as-you-go" assure que nous ne payons que pour les ressources réellement consommées, éliminant les coûts superflus. [cite: 438]

Technologies Utilisées

[cite_start]AWS (DynamoDB, Lambda, SQS, S3, API Gateway), AWS CDK, Python, Git, Jenkins, CI/CD, Infrastructure as Code (IaC). [cite: 21, 454]
